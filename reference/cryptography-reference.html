1 SHA — Cryptographic Hash Function

Compresses any input into a fixed-size digest. Three properties make it useful for security:

Properties
  • Deterministic — same input → same output always
  • One-way — cannot reverse a hash to get the input
  • Avalanche effect — one byte changes → completely different output
  • Collision resistant — can't find two inputs with same hash
Sizes
  • SHA-1 — 160-bit · 40 hex chars · older, weakening
  • SHA-256 — 256-bit · 64 hex chars · modern standard

Two roles SHA plays:
  • Security primitive — compresses data before signing
  • Identifier — fingerprint = short name for a certificate

→ SHA-256 →

"Hello, World."

2 Asymmetric Key Pair

🔑 Private Key — keep secret
  • Never shared, never leaves your machine
  • Encrypts things the public key can decrypt — called signing
  • Decrypts things encrypted with the public key — called decryption
🔓 Public Key — share freely
  • Safe to publish, embed in APKs, send to servers
  • Decrypts things encrypted with the private key — called verifying
  • Encrypts things only the private key can decrypt — called encryption

Direction 2 — Confidentiality (encryption)

Asymmetric crypto is slow
Never used to encrypt bulk data. TLS uses it only for the handshake (authentication + key exchange), then switches to fast symmetric encryption (AES) for actual data.

3 Certificate Anatomy

An X.509 certificate is a structured file. It contains the public key, proves who it belongs to, and proves it hasn't been tampered with.

SIGNATURE = encrypt( SHA(data block above) , issuer's private key )
Fingerprint is derived FROM the certificate — not stored inside it

Self-signed (Android keystores)
  • You are both issuer and subject
  • Your own private key signs the cert
  • No third party involved
  • Firebase only checks: "does fingerprint match what I registered?"
CA-signed (HTTPS websites)
  • CA (e.g. Let's Encrypt) signs your cert after verifying domain ownership
  • Browser trusts CA (pre-installed root store)
  • Browser transitively trusts your site
  • Needed when there's no prior registration step

The critical distinction
The fingerprint is your identifier (safe to share). The private key is your proof of identity (catastrophic if stolen). Firebase does access control using the fingerprint. Android does cryptographic verification using the full public key already inside the APK.

SSH also authenticates YOU to the server
You generate an SSH key pair with ssh-keygen. Your public key goes on the server (~/.ssh/authorized_keys). At login: server sends you a challenge, you sign it with your private key, server verifies with your stored public key. Two-way — both sides authenticate each other.

Modern TLS 1.3 handshake (ECDHE — has forward secrecy)
1. Client hello   → "I support these cipher suites"
2. Server hello   → sends certificate (signed by CA)
3. Client verifies → CA chain check → extracts server public key
4. Key exchange   → both sides run Diffie-Hellman math → shared secret
                  server signs handshake with private key → client verifies
5. Encrypted      → all data encrypted with shared secret (AES)
Old RSA key exchange (deprecated)
Client encrypts secret with server's public key → server decrypts with private key → shared secret

✗ No forward secrecy — if private key stolen later, all past sessions decryptable
Modern ECDHE (TLS 1.3)
Both sides compute shared secret via math (secret never transmitted) → server signs to prove identity

✓ Forward secrecy — private key theft later can't decrypt past sessions

The pattern that repeats everywhere
Fingerprint = short human-readable identifier. It is never the cryptographic mechanism itself.
The real proof is always: sign with private key → verify with public key.
SHA appears in two roles: as a compression step before signing, and as a convenient short name for a public key / certificate.

8 Glossary

TermDefinition
SHASecure Hash Algorithm. Converts any data to a fixed-size digest. One-way, deterministic, avalanche effect.
Digest / HashThe output of a hash function. Fixed size regardless of input size.
FingerprintSHA hash of a certificate. A short, unique identifier for that certificate. Not a security mechanism — a convenience identifier.
Private keyThe secret half of a key pair. Used to sign (prove identity) or decrypt. Must never leave your control.
Public keyThe shareable half of a key pair. Used to verify signatures or encrypt messages. Safe to publish.
Certificate (X.509)A file containing a public key + metadata (owner, expiry, issuer) + a signature over that data. Proves the public key belongs to the stated owner.
Self-signed certificateA certificate signed by its own private key. No CA involved. Used in Android keystores and development environments.
CA (Certificate Authority)A trusted third party (e.g. Let's Encrypt) that signs certificates after verifying the requester owns the domain. Solves the "who vouches for you?" problem.
KeystoreA password-protected file containing your private key and certificate. Generated by keytool. Must be backed up and kept secret.
APK signatureencrypt(SHA(APK bytes), private key). Stored inside the APK. Lets Android verify the APK was not modified after signing.
SSH host keyA key pair generated when a server OS boots. The server signs challenges with it to prove it is the real server, not an impersonator.
known_hostsA file on your machine storing SSH servers' public keys. Used to detect if a server's key changes (MITM or rebuild).
Forward secrecyProperty of ECDHE: the session secret is ephemeral and never transmitted. Stealing the private key later cannot decrypt past sessions.
TLS handshakeThe setup phase of an HTTPS connection. Uses asymmetric crypto to authenticate the server and derive a shared symmetric key.
mTLSMutual TLS. Both client and server present certificates. Used in APIs and service-to-service auth.
