Public-Key Cryptography: The Root of Everything

Answers your 4 questions about keytool, SSH, MITM, and certificates — all from one idea.

Your 4 questions
  1. Is a fingerprint needed to SSH into a GCP machine?
  2. Where does our certificate come from?
  3. What if a man-in-the-middle steals my certificate and presents it?
  4. What is the keytool command doing — is it generating a public and private key?
All four have the same underlying answer: public-key cryptography.

The padlock analogy

Imagine a special padlock that anyone can close, but only you can open. You hand out copies of this open padlock to everyone. Anyone can put a message in a box, snap your padlock shut, and send it to you. Only you — holding the one true key — can open it.

🔓 Public key — the open padlock. Share it with the world. Safe to publish.
🔑 Private key — the key that opens it. Keep it secret. Never share it.

Together they are a key pair. Mathematically linked. Generated together.
You can derive nothing useful about one from the other.

This is public-key cryptography (also called asymmetric cryptography). The two operations it enables are:

  • Encryption: anyone encrypts with your public key → only you can decrypt (private key).
  • Signing: you sign with your private key → anyone can verify with your public key.

Android APK signing and SSH both use the signing direction.

Question 4: What is keytool doing?

Yes — keytool -genkeypair generates a key pair. Here is exactly what happens:

keytool -genkeypair \
  -alias androiddebugkey \
  -keyalg RSA \
  -keysize 2048 \
  -keystore debug.keystore
Step 1 Generate a private key (RSA 2048-bit, stored encrypted in the keystore file)
Step 2 Derive the matching public key (mathematically linked to the private key)
Step 3 Wrap the public key into an X.509 certificate (a standard envelope that adds: owner name, expiry date, issuer, algorithm info)
Step 4 Self-sign that certificate using the private key (proves it hasn't been tampered with)
Step 5 Store everything in the keystore file (.jks / .keystore)

The keystore is just a password-protected file that holds your private key and its certificate. The fingerprint from Lesson 1 is the SHA hash of that certificate (Step 3/4 above).

Key insight
The certificate is public. You can hand it to anyone — that's literally what you do when you paste the fingerprint into Firebase. The private key is what must never leave your machine.

Question 2: Where does the certificate come from?

For Android apps, you create it yourself with keytool (as above). It is self-signed — meaning you are both the issuer and the subject. There is no third party involved.

This is different from HTTPS website certificates, where a Certificate Authority (CA) like Let's Encrypt or DigiCert signs your certificate — vouching that you own the domain. Android doesn't need a CA because Firebase doesn't care who you are in the real world; it only cares that the APK was signed by the same key you registered. Identity here is "matches the registered fingerprint," not "verified by a trusted third party."

HTTPS certificate: your key → signed by → Let's Encrypt (CA) → trusted by browsers
Android keystore: your key → self-signed → registered in Firebase console → trusted by Firebase

Both use the same public-key infrastructure — different trust anchors.

Question 3: What if a MITM steals my certificate?

They get nothing useful. The certificate contains only the public key. It is already public — Firebase shows it in the console, your APK ships it to every user's phone. There is no secret in the certificate.

The real threat is someone stealing your private key. With it, they could:

  • Sign a malicious APK that appears to come from you
  • Firebase would accept it (same fingerprint → same key → trusted)
  • Google Play would accept it as an update to your app
Attacker gets your certificate (public key) → can verify your signatures, but cannot create new ones → harmless

Attacker gets your private key → can sign anything and impersonate you completely → catastrophic
Why losing a private key is an emergency
If your Android release keystore is stolen, you must: revoke it with Google Play (if using Play App Signing), remove the fingerprint from Firebase, generate a new key, re-release the app. Users on the old version can no longer receive updates from you. This is why the keystore file should be in a password manager or secret vault — never in git.

Question 1: SSH into a GCP VM — is a fingerprint involved?

Yes, but which direction is reversed. SSH uses the exact same public-key cryptography, applied twice — once for you proving yourself to the server, once for the server proving itself to you.

You → Server (authentication)

You generate an SSH key pair with ssh-keygen. Your public key goes into the VM (via GCP Console, metadata, or OS Login). When you connect, your SSH client signs a challenge with your private key — the server verifies with your public key. No password needed.

Server → You (host verification)

The VM also has its own key pair (the host key), generated when the OS boots. The first time you connect, SSH shows you the SHA-256 fingerprint of the server's host key: 

The authenticity of host '34.x.x.x' can't be established.
ED25519 key fingerprint is SHA256:AbCdEf1234...
Are you sure you want to continue connecting? (yes/no)?

You type yes → SSH saves it in ~/.ssh/known_hosts. Next connection: SSH checks the fingerprint silently. If it changed (VM rebuilt, or MITM attack), SSH refuses and shows a loud warning.

Android signing → you prove APK identity to Firebase (your app's cert fingerprint, registered in advance)
SSH host verification → server proves identity to you (server's host key fingerprint, stored in known_hosts)
SSH auth → you prove identity to server (your SSH public key, stored on the VM)

GCP's gcloud compute ssh handles host key verification automatically and can show you a VM's host fingerprints via the Console (VM details → SSH Keys tab). You don't register a fingerprint to SSH into GCP — you add your SSH public key to the VM.

The unified picture

Every scenario you have asked about uses the same 3-step pattern:

1. Generate a key pair (private key + public key)
2. Share the public key with the party that needs to verify you (Firebase console / VM authorized_keys / HTTPS CA)
3. Sign something with the private key at runtime (APK bytes / SSH handshake challenge / TLS handshake)

The fingerprint is always the SHA hash of the public key (or its certificate) — a short, shareable identifier for "which key pair are we talking about."

What you just learned
Public-key cryptography is two linked keys: private (sign / keep secret) and public (verify / share freely). keytool generates both and wraps the public key in a self-signed certificate. The fingerprint is the SHA hash of that certificate. A MITM stealing the certificate gets nothing — the private key is the secret. SSH uses the same idea in both directions: you authenticate to the server, and the server authenticates to you.
