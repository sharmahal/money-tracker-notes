Firebase Setup: The Complete Picture

From console to code — how this project connects to Firebase, and why each piece exists.

The one-line mental model

Firebase is a Backend-as-a-Service (BaaS). Google runs the servers. You write no backend code — you just call their SDK from your app. Two services matter for this project:

ServiceWhat it doesAnalogy
Firebase AuthIdentity — who is this user?The bouncer at the door
FirestoreNoSQL cloud database — store and sync dataThe filing cabinet behind the door

The SHA-1 fingerprint you registered (covered in Lesson 1) is how Firebase verifies that requests come from your app — the one signed with your key — and not an impersonator.

Part 1 — The seven setup steps (cloud side)

These are done once per Firebase project in console.firebase.google.com:

  1. Create a Firebase project Just a name and a Google project ID. Nothing installed yet.
  2. Add an Android app → download google-services.json Enter your package name (e.g. com.example.money_tracker). Firebase generates a config file. Put it in android/app/. It contains your API keys and project IDs — Firebase reads it at build time.
  3. Enable Auth → Google sign-in provider In the console: Authentication → Sign-in method → Google → Enable. Without this, Auth calls fail even if your code is correct.
  4. Create a Firestore database Firestore → Create database → choose a region. Start in test mode or production mode (you'll lock it down with security rules next).
  5. Set Firestore security rules Without rules, anyone who gets your app ID can read every user's data. See Part 4 of this lesson for the rule pattern.
  6. Add your SHA-1 fingerprint Project settings → Your apps → Add fingerprint. This is the identity check for Google Sign-In. Debug key: ~/.android/debug.keystore. Release key: your release keystore.
  7. Run flutterfire configure A CLI tool (dart pub global activate flutterfire_cli). This one command does four things automatically: downloads google-services.json → android/app/, edits android/settings.gradle.kts to declare the Gradle plugin, edits android/app/build.gradle.kts to apply it, and generates lib/firebase_options.dart.
Why two config files?

google-services.json is the Android build system's config — Gradle reads it at compile time to embed your Firebase IDs into the APK.
firebase_options.dart is the Dart/Flutter config — Firebase.initializeApp() reads it at runtime to know which project to connect to.

They contain overlapping information but serve different layers (build system vs runtime SDK).

Part 2 — Flutter project setup (your code)

pubspec.yaml — four packages

pubspec.yaml
firebase_core: ^3.0.0      # always required — initialises the SDK
firebase_auth: ^5.0.0      # login / logout / current user
cloud_firestore: ^5.0.0    # read and write the database
google_sign_in: ^6.1.0     # Google picker UI (separate from Firebase Auth)
Why is google_sign_in separate?
Firebase Auth handles the Firebase identity layer. But showing the Google account picker is a Google-specific UI that lives in its own package. You need both: google_sign_in to get the Google credential, then firebase_auth to convert it into a Firebase session.

Android build files — registering the Gradle plugin

No manual editing needed
flutterfire configure (step 7 above) makes these Gradle edits automatically. The files below show what it writes — useful for understanding or debugging, not for copy-pasting.
android/settings.gradle.kts — written by flutterfire configure
// Register the plugin (but don't apply it yet)
id("com.google.gms.google-services") version "4.4.2" apply false
android/app/build.gradle.kts — written by flutterfire configure
// Apply it to the app module — this is what reads google-services.json
id("com.google.gms.google-services")

The Google Services Gradle plugin processes google-services.json and injects the values into your Android build as string resources and metadata. Without it, Firebase can't find your project ID at runtime on Android.

main.dart — always the first thing

lib/main.dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized(); // required before any async setup
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform, // from firebase_options.dart
  );
  runApp(const MyApp());
}

Firebase.initializeApp() must complete before any Firebase SDK call — Auth, Firestore, anything. That's why it's awaited before runApp.

Part 3 — The Auth flow

Auth is split across two layers. AuthService is the low-level Firebase wrapper. AppProvider is the state manager that the UI talks to — it never calls AuthService directly.

UI (HomeScreen, etc.)
  ↓ context.read<AppProvider>().signInWithGoogle()
AppProvider         ← holds _currentUser, _syncing, _lastSyncedAt
  ↓ AuthService.signInWithGoogle()
AuthService         ← low-level Firebase calls only
  ↓ FirebaseAuth + GoogleSignIn SDK
Firebase (cloud)

AuthService — the Firebase wrapper

All methods are static. It does nothing except talk to Firebase and return results.

lib/services/auth_service.dart
// Step 1: Open the Google account picker → get a Google credential
final googleUser = await _googleSignIn.signIn();
final googleAuth = await googleUser!.authentication;

// Step 2: Convert to a Firebase credential → log in
final credential = GoogleAuthProvider.credential(
  accessToken: googleAuth.accessToken,
  idToken: googleAuth.idToken,
);
final result = await _auth.signInWithCredential(credential);
return result.user; // returns the Firebase User, or null on failure

// Sign out — must sign out of both Google and Firebase
await _googleSignIn.signOut();
await _auth.signOut();

AppProvider — the state layer

AppProvider holds all the UI-visible auth state and reacts to auth changes via a stream subscription started in its constructor.

lib/providers/app_provider.dart — constructor
User? _currentUser;
bool _syncing = false;
DateTime? _lastSyncedAt;
StreamSubscription<User?>? _authSub;

AppProvider() {
  // Subscribe to Firebase's auth state stream.
  // Fires immediately on startup (restores saved session) and on every
  // login/logout thereafter.
  _authSub = AuthService.authStateChanges.listen((user) {
    _currentUser = user;
    if (user != null) _fetchLastSyncedAt(); // load last sync time from Firestore
    notifyListeners(); // rebuilds any widget watching AppProvider
  });
}

@override
void dispose() {
  _authSub?.cancel(); // prevent memory leak when provider is removed
  super.dispose();
}
Why a stream, not a one-time call?

authStateChanges() fires automatically on three occasions: (1) when the app starts and Firebase restores a saved session, (2) when the user signs in, (3) when the user signs out. Using a stream means AppProvider is always in sync with Firebase's auth state without any polling.

lib/providers/app_provider.dart — sign in / sign out
// Thin wrapper — delegates to AuthService, returns bool for the UI
Future<bool> signInWithGoogle() async {
  final user = await AuthService.signInWithGoogle();
  return user != null;
  // Note: _currentUser is NOT set here. The authStateChanges stream
  // fires automatically after sign-in and sets it.
}

Future<void> signOut() async {
  await AuthService.signOut();
  _lastSyncedAt = null; // clear sync timestamp from UI
  notifyListeners();
  // Again, _currentUser is cleared by the authStateChanges stream.
}

Part 4 — Firestore: data model and patterns

Firestore is not SQL. It's a hierarchy: collections → documents → fields.

// This is the structure used in this project
users/ ← collection
  {uid}/ ← document (one per logged-in user)
    name: "Nishant" ← fields
    email: "..."
    months/ ← subcollection
      2025-06/ ← document per month
        transactions: [] ← array field
      2025-05/
        transactions: []
lib/services/sync_service.dart — write
final db = FirebaseFirestore.instance;

// Overwrite the whole document
await db.collection('users').doc(uid).set({'name': 'Nishant'});

// Merge — only update listed fields, leave others alone
await db.collection('users').doc(uid).set(
  {'lastSeen': Timestamp.now()},
  SetOptions(merge: true),
);
lib/services/sync_service.dart — read
// Read once
final doc = await db.collection('users').doc(uid).get();
final data = doc.data(); // Map<String, dynamic>? — null if doc doesn't exist

// Read a subcollection (all months)
final snap = await db
    .collection('users').doc(uid)
    .collection('months').get();
for (final doc in snap.docs) {
  print(doc.id);     // e.g. "2025-06"
  print(doc.data()); // the fields
}

// Real-time listener (fires every time data changes on the server)
db.collection('users').doc(uid).snapshots().listen((snap) {
  print(snap.data());
});

Security rules — the gatekeeper

Firestore console → Rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{uid}/{document=**} {
      allow read, write: if request.auth != null
                         && request.auth.uid == uid;
    }
  }
}

Two checks: (1) the request must be authenticated (logged in), and (2) the Firebase UID of the logged-in user must match the {uid} in the document path. This means each user can only read and write their own subtree. Without this, anyone with your app's credentials could read anyone's data.
